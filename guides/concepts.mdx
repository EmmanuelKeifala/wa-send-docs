## How to read the API reference

The fastest way to move through wa-send is to map each concept to the exact endpoint that implements it.

- API key creation -> `POST /api/keys`
- template management -> `GET /api/templates`, `POST /api/templates`, `PUT /api/templates/{id}`
- message sending -> `POST /api/messages`

Once you understand the concepts on this page, move directly into the matching endpoint pages in the [API Reference](../api-reference).

## The account model

Your account is where you manage the resources that shape your integration:

- API keys for authenticated server requests
- templates for reusable message content
- profile settings for account-level configuration

## Two authentication models

wa-send uses two kinds of credentials:

- `Authorization: Bearer <jwt>` for signed-in, account-scoped actions
- `x-api-key: <api-key>` for server-side message submission

In practice, teams usually sign in once to create templates and keys, then use API keys in production systems.

Relevant endpoints:

- `POST /api/keys`

## Direct messages vs templates

You can send messages in two ways:

- direct messages when the copy is assembled in your application
- templates when the message structure should be reusable and centrally managed

Direct messages are useful for simple flows. Templates are better when you want consistency across products, teams, or environments.

Recommended default:

- use templates for recurring product flows
- use direct messages for one-off or early-stage internal integrations

Relevant endpoints:

- `POST /api/messages`
- `GET /api/templates`
- `POST /api/templates`

## Environments

API keys are created in one of two environments:

- `test`
- `live`

This makes it easier to separate development traffic from production traffic while keeping the request format the same.

Relevant endpoint:

- `POST /api/keys`

## Accepted vs delivered

When wa-send returns a successful response for a message request, it is confirming that the platform accepted the request.

That means:

- the payload was valid
- authentication passed
- the message entered the platform for processing

For most developers, this is the right contract: your application can continue without waiting on the full delivery lifecycle.

Relevant endpoint:

- `POST /api/messages`

## A practical integration pattern

Most teams follow this sequence:

1. create an account
2. create a `test` API key
3. send direct messages to validate the flow
4. move common copy into templates
5. switch to a `live` key when ready to launch

This is the same pattern used by the nicer published docs site, but here it stays aligned to the real backend routes and request validation rules.
