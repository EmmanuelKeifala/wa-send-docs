## Send Message

**Method:** `POST`

**Route:** `/api/messages`

Use this endpoint for both direct message sends and template-based transactional flows.

## What is covered

- direct and template-based message submission
- template CRUD
- set default template
- API key management
- profile read and update
- account deletion
- service health

## Core integration routes

| Route | Auth | Purpose |
| --- | --- | --- |
| `POST /api/messages` | `x-api-key` or bearer token | Queue a direct message or resolve and queue a template-based message |
| `GET /api/templates` | bearer token | List templates owned by the current account |
| `POST /api/templates` | bearer token | Create a reusable template |
| `PUT /api/templates/{id}` | bearer token | Update an existing template |
| `PUT /api/templates/{id}/set-default` | bearer token | Mark one template as the default for the account |
| `DELETE /api/templates/{id}` | bearer token | Delete a template |
| `GET /api/keys` | bearer token | List issued API keys |
| `POST /api/keys` | bearer token | Create a `test` or `live` API key |
| `DELETE /api/keys/{id}` | bearer token | Revoke an API key |

## Messaging request body

`POST /api/messages` accepts one of these request shapes:

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `to` | string | Yes | Must match `^\+?[1-9]\d{1,14}$` |
| `message` | string | Conditionally | Required for direct sends. Do not send it with `templateId`. |
| `templateId` | string | Conditionally | Required for template sends. Do not send it with `message`. |
| `templateArgs` | object | Conditionally | Required when `templateId` is present. Keys should match placeholders like `{{otp}}`. |

Template argument keys are backend-defined by your template content. Common placeholders are `otp`, `expires`, `customer_name`, and `order_id`.

## Response

Successful requests return `202 Accepted` when the message has been validated and queued.

```json
{
  "id": "queue-message-id",
  "status": "queued",
  "timestamp": "2026-03-28T12:00:00.000Z"
}
```

Error responses follow the same basic shape:

```json
{
  "code": "VALIDATION_ERROR",
  "error": "Invalid phone number format",
  "details": [
    {
      "field": "to",
      "message": "Invalid phone number format"
    }
  ]
}
```

The stable integration error contract is:

- HTTP status code
- machine-readable `code`
- human-readable `error`
- validation `details` when the request body is invalid

Valid examples:

```json
{
  "to": "+23276123456",
  "message": "Your verification code is 483920"
}
```

```json
{
  "to": "+23276123456",
  "templateId": "template-uuid",
  "templateArgs": {
    "otp": "483920",
    "expires": "5"
  }
}
```

## Error codes

| Canonical code | Status | Actual backend message | Recommended action |
| --- | --- | --- | --- |
| `VALIDATION_ERROR` | `400` | `Either "message" or "templateId" must be provided` | fix the request and retry |
| `AUTH_REQUIRED` | `401` | `Authentication required. Please provide x-api-key header or Bearer token.` | send valid credentials |
| `INVALID_API_KEY` | `401` | `Invalid API Key` | rotate or replace the API key |
| `API_KEY_INACTIVE` | `401` | `API Key is inactive` | reactivate or replace the key |
| `INVALID_TOKEN` | `401` | `Invalid or expired token. Please login again.` | refresh the dashboard session |
| `USER_NOT_FOUND` | `401` | `User not found. Please login again.` | sign in again with a valid account |
| `TEMPLATE_NOT_FOUND` | `404` | `Template not found` | verify the template ID and ownership |
| `TEMPLATE_RESOLUTION_FAILED` | `500` | `Failed to resolve template` | retry only if the failure is transient |
| `AUTHENTICATION_ERROR` | `500` | `Authentication error` | inspect auth configuration and retry carefully |
| `INTERNAL_SERVER_ERROR` | `500` | `Internal server error` | retry only if your flow supports transient failures |

| `AUTH_REQUIRED` | protected routes | auth header or API key missing |

## Option 1: Using templates (Recommended)

Templates are the cleaner long-term pattern for OTP, recovery, and notification flows because they keep reusable message copy out of application code.

### cURL

```bash
curl -X POST https://api.wa-send.dev/api/messages \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "to": "+23276123456",
    "templateId": "template-verification-id",
    "templateArgs": {
      "otp": "483920",
      "expires": "5"
    }
  }'
```

### JavaScript

```js
const response = await fetch('https://api.wa-send.dev/api/messages', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'x-api-key': process.env.WASEND_API_KEY,
  },
  body: JSON.stringify({
    to: '+23276123456',
    templateId: 'template-verification-id',
    templateArgs: {
      otp: '483920',
      expires: '5',
    },
  }),
})

const data = await response.json()
if (!response.ok) throw new Error(data.error || 'Request failed')
```

### Python

```python
import os
import requests

response = requests.post(
    'https://api.wa-send.dev/api/messages',
    headers={
        'Content-Type': 'application/json',
        'x-api-key': os.environ['WASEND_API_KEY'],
    },
    json={
        'to': '+23276123456',
        'templateId': 'template-verification-id',
        'templateArgs': {
            'otp': '483920',
            'expires': '5',
        },
    },
    timeout=30,
)

data = response.json()
response.raise_for_status()
```

### Go

```go
payload := map[string]any{
    "to":         "+23276123456",
    "templateId": "template-verification-id",
    "templateArgs": map[string]string{
        "otp":     "483920",
        "expires": "5",
    },
}
```

## Template request body

Use this shape with `POST /api/templates` and `PUT /api/templates/{id}`.

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `name` | string | Yes | 1 to 100 characters |
| `template_type` | string | Yes | One of `otp`, `verification`, `custom`, `login`, `password_reset`, `transaction` |
| `message_template` | string | Yes | 1 to 1000 characters |
| `otp_length` | integer | No | 4 to 8, defaults to `6` |
| `expires_in` | integer | No | 60 to 3600 seconds, defaults to `300` |
| `is_default` | boolean | No | Optional on create and update |

## Authentication models

- `Authorization: Bearer <jwt>` for account-scoped routes
- `x-api-key: <api-key>` for server-side message submission

## Best practices

- prefer templates for recurring product flows
- include an explicit expiry window in OTP-style messages
- tell users not to share the code
- use `test` keys during development and `live` keys in production
- treat `202 Accepted` as queued, not final delivery confirmation
- separate authentication failures from template lookup failures in your logging

## Rate limits

This backend does not currently publish a dedicated rate-limit contract or `429` response shape in the application routes.

For production integrations:

- avoid bursty retry loops
- add your own backoff around transient `500` responses
- keep OTP retry frequency under your own application control

## SDKs and libraries

- use the bundled TypeScript SDK in `sdk/typescript` if you want a typed client in Node.js
- use plain `fetch`, `requests`, or the Go `net/http` package for lightweight integrations
- use bearer-authenticated routes for template and key management

## Troubleshooting

### Messages are accepted but not yet delivered

`202 Accepted` means the backend validated the request and queued the message. It does not mean the delivery lifecycle is complete.

### Getting `Template not found`

- verify the template ID
- confirm the template belongs to the authenticated account
- make sure you are not mixing credentials across different accounts

### Getting authentication errors

- confirm you are sending either `x-api-key` or `Authorization: Bearer <jwt>`
- verify the API key is active
- verify the bearer token is current and belongs to an existing user

## Advanced

- use `PUT /api/templates/{id}/set-default` to keep a single preferred template for a common flow
- use JWT auth when managing account resources such as templates, keys, and profiles
- use `DELETE /api/profiles` carefully because it removes the current user profile and owned resources
